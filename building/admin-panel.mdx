## Overview

Faster CMS provides two approaches to build admin panels for your plugin:

1. **Schema-Based Admin** - Define data models using a declarative schema (quickest, recommended for CRUD operations)
2. **Custom UI Admin** - Build fully custom React interfaces (maximum flexibility for complex UIs)

<Info>
  Schema-based admin automatically generates CRUD interfaces, while custom UI gives you complete control over the interface.
</Info>

## How Faster CMS Determines Admin Type

Faster CMS determines whether to use schema-based or custom UI based on the `type` attribute in your admin schema.

### Schema-Based UI (Default)

By default, all models use auto-generated schema-based UI:

```javascript schema/admin-schema.js
const modelSchemas = {
    categories: {
        name: 'categories',
        displayName: 'Categories',
        tableName: 'my_plugin_categories',
        icon: 'üìÅ',
        fields: [ /* field definitions */ ],
        actions: ['create', 'read', 'update', 'delete'],
        listView: {
            columns: ['name', 'slug', 'is_active'],
            // No type specified = auto-generated UI
        },
        formView: {
            sections: [ /* sections */ ]
            // No type specified = auto-generated UI
        }
    }
};
```

**Result**: Faster CMS auto-generates list and form interfaces

### Custom UI

To use custom React components, add `type: 'custom'` to `listView` and/or `formView`:

```javascript schema/admin-schema.js
const modelSchemas = {
    orders: {
        name: 'orders',
        displayName: 'Orders',
        tableName: 'my_plugin_orders',
        icon: 'üõí',
        fields: [ /* field definitions */ ],
        actions: ['read', 'update'],
        listView: {
            type: 'custom',  // Use custom Listing.tsx component
            columns: ['order_number', 'customer_email', 'total', 'status'],
        },
        formView: {
            type: 'custom',  // Use custom CreateOrEdit.tsx component
            sections: [ /* sections */ ]
        }
    }
};
```

**Result**: Faster CMS looks for custom components at:
- `admin/orders/Listing.tsx` (for list view)
- `admin/orders/CreateOrEdit.tsx` (for form view)

### Mix and Match

You can use custom UI for list view and schema-based for form view (or vice versa):

```javascript
listView: {
    type: 'custom',  // Custom listing
    columns: ['name', 'status']
},
formView: {
    // No type = auto-generated form
    sections: [ /* sections */ ]
}
```

## Schema-Based Admin

Schema-based admin uses a declarative configuration to automatically generate admin interfaces for your data models.

### Directory Structure

```bash
your-plugin/
‚îú‚îÄ‚îÄ schema/
‚îÇ   ‚îî‚îÄ‚îÄ admin-schema.js       # Admin model definitions
‚îî‚îÄ‚îÄ index.js                  # Plugin entry point
```

### Basic Schema Example

```javascript schema/admin-schema.js
const modelSchemas = {
    items: {
        name: 'items',
        displayName: 'Items',
        tableName: 'my_plugin_items',
        icon: 'üì¶',
        fields: [
            {name: 'id', type: 'string', primary: true, hidden: true},
            {name: 'name', type: 'string', required: true, label: 'Item Name', maxLength: 255},
            {name: 'description', type: 'text', label: 'Description'},
            {name: 'price', type: 'number', format: 'currency', required: true, label: 'Price', min: 0},
            {name: 'is_active', type: 'boolean', label: 'Active', default: true},
            {name: 'created_at', type: 'datetime', hidden: true, auto: true},
            {name: 'updated_at', type: 'datetime', hidden: true, auto: true}
        ],
        actions: ['create', 'read', 'update', 'delete'],
        listView: {
            columns: ['name', 'price', 'is_active'],
            searchable: ['name', 'description'],
            sortable: ['name', 'price', 'created_at'],
            defaultSort: {field: 'name', order: 'asc'}
        },
        formView: {
            sections: [
                {
                    title: 'Basic Information',
                    fields: ['name', 'description']
                },
                {
                    title: 'Pricing',
                    fields: ['price', 'is_active']
                }
            ]
        }
    }
};

module.exports = {
    modelSchemas
};
```

## Model Schema Attributes

### Top-Level Attributes

| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | Yes | Model identifier (lowercase, no spaces) |
| `displayName` | string | Yes | Human-readable name shown in admin |
| `tableName` | string | Yes | Database table name |
| `icon` | string | No | Emoji or icon for the model |
| `fields` | array | Yes | Array of field definitions |
| `actions` | array | No | Allowed actions: `create`, `read`, `update`, `delete` |
| `listView` | object | No | List view configuration |
| `formView` | object | No | Form view configuration |
| `validations` | object | No | Custom validation rules |

## Field Attributes

| Attribute | Type | Description |
|-----------|------|-------------|
| `name` | string | Field identifier (matches database column) |
| `type` | string | Field type (string, number, boolean, select, etc.) |
| `label` | string | Display label in forms |
| `required` | boolean | Whether field is required |
| `primary` | boolean | Mark as primary key |
| `hidden` | boolean | Hide from forms (still in database) |
| `readOnly` | boolean | Display only, not editable |
| `readOnlyOnEdit` | boolean | Read-only when editing (editable on create) |
| `unique` | boolean | Enforce unique constraint |
| `default` | any | Default value for new records |
| `placeholder` | string | Placeholder text |
| `help` | string | Help text shown below field |
| `maxLength` | number | Maximum string length |
| `min` | number | Minimum numeric value |
| `max` | number | Maximum numeric value |
| `pattern` | string | Regex pattern for validation |
| `format` | string | Special format (currency, email, etc.) |
| `unit` | string | Unit label (minutes, kg, etc.) |
| `auto` | boolean | Auto-generated value |
| `virtual` | boolean | Not stored in database |
| `nullable` | boolean | Allow null values |

## Conditional Fields

Show fields based on other field values:

```javascript
{
    name: 'discount_amount',
    type: 'number',
    label: 'Discount Amount',
    conditional: {field: 'has_discount', value: true},
    help: 'Only shown when discount is enabled'
}

// Conditional by category type
{
    name: 'event_duration',
    type: 'number',
    label: 'Duration (minutes)',
    conditional: {categoryType: 'event'},
    min: 5
}
```

## List View Configuration

```javascript
listView: {
    type: 'custom',  // Optional: Use 'custom' for custom UI, omit for auto-generated UI
    columns: ['name', 'email', 'status', 'created_at'],
    searchable: ['name', 'email', 'description'],
    sortable: ['name', 'email', 'created_at'],
    defaultSort: {field: 'created_at', order: 'desc'},
    filters: ['status', 'category_id']
}
```

| Attribute | Type | Description |
|-----------|------|-------------|
| `type` | string | `'custom'` to use custom Listing.tsx component, omit for auto-generated UI |
| `columns` | array | Fields to display in table |
| `searchable` | array | Fields to include in search |
| `sortable` | array | Fields that can be sorted |
| `defaultSort` | object | Default sort configuration |
| `filters` | array | Fields to show as filters |

## Custom UI Admin

For complex interfaces beyond CRUD operations, create custom React components.

<Warning>
  **Important Constraints**:
  - Relative imports are NOT supported - write all code in a single file
  - Only two components are supported: `Listing.tsx` and `CreateOrEdit.tsx`
  - Only Ant Design components and Ant Design icons are available
  - Use injected `window.axiosReq` for API calls
  - Components are served via API and rendered dynamically by Faster backend
</Warning>

### Directory Structure

```bash
your-plugin/
‚îú‚îÄ‚îÄ admin/
‚îÇ   ‚îî‚îÄ‚îÄ items/
‚îÇ       ‚îú‚îÄ‚îÄ Listing.tsx        # List view (required)
‚îÇ       ‚îî‚îÄ‚îÄ CreateOrEdit.tsx   # Create/Edit form (required)
‚îî‚îÄ‚îÄ index.js
```

### Listing Component

The `Listing.tsx` component displays a table of records with actions.

### CreateOrEdit Component

The `CreateOrEdit.tsx` component handles both creating new records and editing existing ones.

```typescript admin/items/CreateOrEdit.tsx
import React, { useEffect } from 'react';
import { Form, Input, InputNumber, Switch, Button, Space, message } from 'antd';

// All types must be defined in the same file
interface CreateOrEditProps {
  pluginId: string;
  mode: 'create' | 'edit';
  initialValues?: Record<string, any>;
  onSubmit: (values: Record<string, any>) => Promise<void>;
  onCancel: () => void;
  loading?: boolean;
}

const CreateOrEdit: React.FC<CreateOrEditProps> = ({
  pluginId,
  mode,
  initialValues,
  onSubmit,
  onCancel,
  loading,
}) => {
  const [form] = Form.useForm();

  useEffect(() => {
    if (initialValues) {
      form.setFieldsValue(initialValues);
    }
  }, [initialValues, form]);

  const handleSubmit = async (values: any) => {
    try {
      await onSubmit(values);
      message.success(`Item ${mode === 'create' ? 'created' : 'updated'} successfully`);
      form.resetFields();
    } catch (error) {
      message.error(`Failed to ${mode} item`);
      console.error(error);
    }
  };

  return (
    <div className="tw-bg-white tw-p-6 tw-rounded-lg">
      <h2 className="tw-text-xl tw-font-semibold tw-mb-6">
        {mode === 'create' ? 'Create Item' : 'Edit Item'}
      </h2>

      <Form
        form={form}
        layout="vertical"
        onFinish={handleSubmit}
        initialValues={initialValues}
      >
        <Form.Item
          name="name"
          label="Item Name"
          rules={[{ required: true, message: 'Please enter item name' }]}
        >
          <Input placeholder="Enter item name" />
        </Form.Item>

        <Form.Item
          name="description"
          label="Description"
        >
          <Input.TextArea rows={4} placeholder="Enter description" />
        </Form.Item>

        <Form.Item
          name="price"
          label="Price"
          rules={[{ required: true, message: 'Please enter price' }]}
        >
          <InputNumber
            min={0}
            step={0.01}
            prefix="$"
            placeholder="0.00"
            className="tw-w-full"
          />
        </Form.Item>

        <Form.Item
          name="is_active"
          label="Active"
          valuePropName="checked"
        >
          <Switch />
        </Form.Item>

        <Form.Item>
          <Space>
            <Button type="primary" htmlType="submit" loading={loading}>
              {mode === 'create' ? 'Create' : 'Update'}
            </Button>
            <Button onClick={onCancel}>
              Cancel
            </Button>
          </Space>
        </Form.Item>
      </Form>
    </div>
  );
};

export default CreateOrEdit;
```

## API Access in Custom UI

Faster CMS injects `axiosReq` globally for making API calls. Access your plugin's backend endpoints using this function.

### Using axiosReq

```typescript
// axiosReq is injected by Faster and available globally
// @ts-ignore
const axiosReq = window.axiosReq;

// GET request - Fetch all items
const response = await axiosReq(
  'GET',
  `studio/cms/plugins/${pluginId}/admin/data/items`
);
const items = response.data;

// POST request - Create new item
await axiosReq(
  'POST',
  `studio/cms/plugins/${pluginId}/admin/data/items`,
  {
    name: 'New Item',
    price: 99.99,
    is_active: true
  }
);

// PUT request - Update existing item
await axiosReq(
  'PUT',
  `studio/cms/plugins/${pluginId}/admin/data/items/${itemId}`,
  {
    name: 'Updated Name',
    price: 149.99
  }
);

// DELETE request - Delete item
await axiosReq(
  'DELETE',
  `studio/cms/plugins/${pluginId}/admin/data/items/${itemId}`
);
```

### API Call Pattern

All API calls should be wrapped in try-catch blocks:

```typescript
const handleDelete = async (record: any) => {
  try {
    // @ts-ignore
    const axiosReq = window.axiosReq;

    await axiosReq(
      'DELETE',
      `studio/cms/plugins/${pluginId}/admin/data/items/${record.id}`
    );

    message.success('Item deleted successfully');
    onRefresh(); // Refresh the list
  } catch (error) {
    console.error('Delete failed:', error);
    message.error('Failed to delete item');
  }
};
```

## Best Practices

<AccordionGroup>
  <Accordion title="Choose the Right Approach" icon="compass">
    - Use **Schema-Based** for standard CRUD operations
    - Use **Custom UI** for complex workflows, custom visualizations, or unique interactions
    - Combine both approaches in the same plugin
    - Schema-based is faster to develop and maintain
  </Accordion>

  <Accordion title="Schema Design" icon="database">
    - Keep field names consistent with database columns
    - Use descriptive labels and help text
    - Group related fields in form sections
    - Add conditional fields to reduce clutter
    - Use appropriate field types and validations
  </Accordion>

  <Accordion title="Custom UI Constraints" icon="code">
    - **No relative imports** - all code must be in a single file
    - Only use Ant Design components and Ant Design icons
    - Only two components supported: `Listing.tsx` and `CreateOrEdit.tsx`
    - Define all types inline in the same file
    - Use injected `window.axiosReq` for API calls
    - Components are rendered dynamically by Faster backend
  </Accordion>

  <Accordion title="Custom UI Performance" icon="bolt">
    - Implement pagination for large datasets
    - Use loading states for async operations
    - Debounce search inputs when needed
    - Keep component code minimal and focused
    - Avoid complex computations in render
  </Accordion>

  <Accordion title="User Experience" icon="user">
    - Provide clear feedback on actions using `message` from Ant Design
    - Use confirmation dialogs for destructive actions (`Popconfirm`)
    - Show loading states during operations
    - Display meaningful error messages
    - Implement search and filtering
  </Accordion>

  <Accordion title="Error Handling" icon="triangle-exclamation">
    - Always wrap API calls in try-catch blocks
    - Display user-friendly error messages with `message.error()`
    - Log errors to console for debugging
    - Provide clear feedback when operations fail
    - Validate data before submission
  </Accordion>
</AccordionGroup>
