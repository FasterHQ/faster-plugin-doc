## Architecture Highlights

### Provider Pattern

```javascript
// Base payment provider
class BaseProvider {
    async createPayment(amount, metadata) {
        throw new Error('Must implement');
    }
}

// Concrete provider
class StripeProvider extends BaseProvider {
    async createPayment(amount, metadata) {
        return await this.stripe.paymentIntents.create({...});
    }
}

// Registry
registry.register('stripe', StripeProvider);
```

### Repository Pattern

```javascript
class ProductRepository {
    async findById(id, relations, transacting) {
        return await Product.findOne({id}, {
            withRelated: relations,
            transacting
        });
    }

    async createWithVariants(data, transacting) {
        // Complex multi-step operation
    }
}
```

### Service Layer

```javascript
class OrderService {
    async createOrder(cartId, memberId, shipping) {
        return await ghostBookshelf.transaction(async (t) => {
            const cart = await this.getCart(cartId, t);
            await this.validateInventory(cart, t);
            const order = await this.createOrderRecord(cart, t);
            await this.updateInventory(cart, t);
            await this.clearCart(cartId, t);
            return order;
        });
    }
}
```

## Key Learnings

<AccordionGroup>
  <Accordion title="Transactions are Critical">
    Always use transactions for multi-step operations to ensure data consistency.
  </Accordion>

  <Accordion title="Service Layer Benefits">
    Separating business logic into services makes code testable and reusable.
  </Accordion>

  <Accordion title="Repository Pattern">
    Abstracting data access allows for easier testing and modifications.
  </Accordion>

  <Accordion title="Provider Pattern for Extensibility">
    The provider pattern makes it easy to add new payment methods without changing core code.
  </Accordion>
</AccordionGroup>
